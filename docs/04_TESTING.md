# 测试策略 (TESTING)

> 本文件定义测试范围、分层、完成定义与每功能测试要求。
> **核心原则：未通过测试的功能 = 未完成的功能。** 无例外。

---

## 1. 测试金字塔

```
        ┌──────────┐
        │ 端到端 E2E │  少量：完整对局流程
       ┌┴──────────┴┐
       │  集成测试    │  中量：模块协作、回合流程、网络
      ┌┴────────────┴┐
      │   单元测试     │  大量：规则校验、状态变更、纯函数
      └──────────────┘
```

- **单元测试**占 ≥70%，覆盖 Layer 1 全部公共 API
- **集成测试**占 ~25%，覆盖 Layer 2/3 协作
- **端到端**占 ≤5%，覆盖完整对局从开局到胜利

## 2. 测试框架

- **GUT (Godot Unit Test)**：单元 + 集成
- **Godot 测试场景**：端到端可编写自动化脚本对局
- 测试位于 `project/tests/`，镜像 `src/` 结构
  - `tests/unit/core/` 对应 `src/core/`
  - `tests/integration/` 跨模块
  - `tests/e2e/` 完整对局

## 3. 命名约定

- 测试文件：`<被测类>_test.gd`（如 `rules_engine_test.gd`）
- 测试方法：`test_<被测行为>_<条件>_<预期>()`
  - 例：`test_apply_build_settlement_when_adjacent_to_own road_succeeds()`
- 测试套件类名：`<被测类>Test`（如 `RulesEngineTest`）

## 4. 分层测试要求

### 4.1 Layer 1 核心层（强制 100% 覆盖公共 API）

每个公共方法必须有：
- **正向用例**：合法输入返回成功
- **负向用例**：每种非法输入各 1 例，验证错误码
- **边界用例**：空、极值、临界条件

#### 必测规则清单（对应 [GAME_RULES.md](01_GAME_RULES.md)）

| 规则 | 必测场景 |
|------|----------|
| 棋盘生成 | 地形分布数量正确；数字牌分布正确；6/8 不相邻（开启平衡时） |
| 初始放置 | 距离规则；道路连接；第二定居点产出初始资源 |
| 掷骰产出 | 7 不产出；匹配数字产出；定居点 1 张/城市 2 张；强盗压制 |
| 强盗 | 手牌 >7 弃半（奇数向下）；移动约束；偷取随机性；目标无资源 |
| 建造 | 各建筑成本校验；位置合法性；道路连接性；定居点距离规则 |
| 交易 | 4:1 银行；3:1 港口；2:1 专项港；港口归属校验；资源不足拒绝 |
| 发展卡 | 购买当回合不可用；每回合限 1 张；骑士/垄断/发明/道路建设/胜利点各自效果 |
| 最长道路 | ≥5 触发；转移；断链；船+路混合 |
| 最大军队 | ≥3 触发；转移 |
| 胜利条件 | 10 点触发；隐藏胜利点卡结算；同时达到裁定 |
| 黄金地形 | 选择资源；城市选 2 张 |
| 船只 | 海洋边建造；可移动；移动约束 |

### 4.2 Layer 2 应用层

- 回合状态机：每个状态合法/非法动作集合
- 事件总线：事件正确分发与订阅
- GameSession：动作提交 → 状态变更 → 事件产出链路

### 4.3 Layer 3 网络层

- 协议序列化/反序列化往返一致
- 服务器校验拒绝非法客户端动作
- 断线重连后快照恢复正确
- AI 玩家接入：与人类客户端走相同校验路径
- **网络测试须模拟多客户端**（可用 mock transport，不依赖真实网络）

### 4.4 Layer 4 表现层

- 表现层测试非强制，但关键交互须有冒烟测试
- 视图正确响应事件总线事件（可 mock 状态）

## 5. 测试数据与确定性

- **禁止依赖随机**：测试中注入固定种子或 mock 随机源
- 提供 `TestFixtures`：预构造棋盘、状态、动作工厂
- 共享 fixture 放 `tests/fixtures/`
- 每个测试独立，不依赖执行顺序，不依赖全局可变状态

## 6. 完成定义 (Definition of Done) — 测试维度

一个功能/任务标记为完成前，必须满足：

- [ ] 对应单元测试已编写且通过
- [ ] 覆盖正向 + 负向 + 边界用例
- [ ] 既有全部测试通过（`gut` 全绿）
- [ ] 无 `pending`/`skip` 的未完成测试（除非显式标注原因 + issue）
- [ ] 测试不依赖网络/外部资源（除非端到端且已隔离）
- [ ] 测试命名符合 §3
- [ ] [PROGRESS.md](05_PROGRESS.md) 中该条目状态更新为 ✅ 并附测试文件路径

## 7. 测试运行

- 本地：`godot --headless -s addons/gut/gut_cmdln.gd -gdir=res://tests`
- CI：每次提交/PR 自动运行，失败阻断合并
- 覆盖率：定期生成报告（GUT 配合 coverage 工具），核心层目标 ≥90%

## 8. 测试反模式（禁止）

- 测试断言"不崩溃"而非具体行为（`assert_not_null` 滥用）
- 一个测试断言过多（>5 个 assert），须拆分
- 测试间共享可变状态
- 为通过测试而修改测试（应修改实现或修正规格）
- Mock 一切导致测试不测真实逻辑
- 跳过失败的测试而非修复

## 9. 缺陷修复流程

发现 bug 时：

1. 先编写**复现测试**（红）
2. 修复实现使测试通过（绿）
3. 补充边界用例
4. 在 [PROGRESS.md](05_PROGRESS.md) 记录 bug 与修复条目
5. 评估是否有类似缺陷需排查

## 10. 测试与规格一致性

- 测试是规格的可执行表达
- [GAME_RULES.md](01_GAME_RULES.md) 每条规则都应有对应测试
- 规则变更须同步变更测试，反之亦然
- 定期审查"有规则无测试"或"有测试无规则"的偏差

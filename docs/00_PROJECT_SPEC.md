# 项目规格说明书 (PROJECT_SPEC)

> 本文件是项目的最高约束文件。所有后续开发、AI 决策、架构选型必须以本文件为准。
> 若与其他文档冲突，以本文件优先级最高。

---

## 1. 项目愿景

制作一款基于 Godot 4.x 的《卡坦岛》(Catan) 桌游电子化实现，供项目所有者个人游玩。

- **核心玩法**：经典卡坦岛基础版 + 海洋扩展(Seafarers)
- **首要目标**：支持服务器联机多人对战（权威服务器模型）
- **次要目标**：支持 AI 对手（在联机稳定后实现）
- **美术风格**：几何色块（纯色多边形 + 图标），不依赖外部美术素材

## 2. 技术栈

| 项目 | 选型 | 说明 |
|------|------|------|
| 引擎 | Godot 4.x 稳定版 | 使用最新稳定 release，不使用 beta/RC |
| 语言 | GDScript 2.0 | 全部业务代码使用 GDScript，不混用 C# |
| 网络 | Godot 高级多人 API | ENetMultiplayerPeer (LAN/直连) + WebSocketPeer (穿透/网页) |
| 测试框架 | GUT (Godot Unit Test) | 单元测试与集成测试统一使用 GUT |
| 数据格式 | JSON / Godot Resource (.tres) | 规则数据、场景配置数据驱动 |
| 版本控制 | Git | 强制使用，每个功能分支独立提交 |

## 3. 范围边界 (Scope)

### 3.1 IN SCOPE（必须实现）

- 基础版完整规则（见 [GAME_RULES.md](01_GAME_RULES.md)）
- 海洋扩展核心机制：船只、海洋地形、岛屿场景、黄金资源
- 至少 2 个海洋扩展场景（推荐：The New World、Into the Desert）
- 服务器权威联机模式（3-4 人房间）
- 房间创建/加入/断线重连基础流程
- AI 对手（中等难度策略，至少 1 种策略实现）
- 几何色块渲染（地形、建筑、资源卡、发展卡）
- 本地热座模式（作为联机开发前的过渡形态）
- 存档/读档（单局中断恢复）
- 完整单元测试 + 集成测试覆盖核心规则

### 3.2 OUT OF SCOPE（明确不实现，禁止越界）

- 城市与骑士扩展 (Cities & Knights)
- 商人与野蛮人扩展 (Traders & Barbarians)
- 题材包、独立扩展场景包
- 排位赛、天梯、匹配系统
- 账号系统、用户登录、好友列表
- 聊天系统、表情、语音
- 移动端原生打包（iOS/Android）、触屏优化
- 3D 渲染、物理引擎、粒子特效
- 自定义规则编辑器（仅保留数据驱动的扩展点，不做玩家可视化编辑）
- 反作弊、服务器端持久化数据库
- 多语言本地化（初期仅中文，但代码须预留 i18n 扩展点）

### 3.3 灰色地带（需逐项确认后才可推进）

下列内容默认不实现，除非在 [PROGRESS.md](05_PROGRESS.md) 中显式标注并由所有者确认：

- 5-6 人扩展（需额外测试平衡与 UI 适配）
- 自定义随机种子回放
- 观战模式
- 录像/复盘导出

## 4. 硬性约束（不可违反的底线）

以下约束任何 AI 开发会话都不得违反，违反即视为开发失败：

1. **测试先行/同步**：每个功能必须伴随测试。功能未通过测试 = 功能未完成。详见 [TESTING.md](04_TESTING.md)。
2. **核心逻辑与表现层分离**：游戏规则引擎必须是纯 GDScript 类（不继承 Node），可在无 Godot 场景环境下被测试。渲染/网络/UI 不得侵入规则引擎。
3. **数据驱动**：地形类型、建筑成本、发展卡效果、场景布局等可配置项必须以数据文件形式存在，禁止硬编码到逻辑分支中。
4. **扩展性优先**：新增规则、场景、AI 策略必须通过注册/插件式扩展点接入，不得修改核心引擎主流程。
5. **网络权威模型**：所有游戏状态变更必须经服务器验证。客户端永远不能直接修改权威状态。
6. **不破坏既有测试**：每次提交前所有既有测试必须通过。新增功能不得删除既有测试。
7. **文档同步**：[PROGRESS.md](05_PROGRESS.md) 必须在每次功能完成后更新状态。架构变更必须同步更新 [ARCHITECTURE.md](02_ARCHITECTURE.md)。
8. **范围纪律**：禁止擅自实现 OUT OF SCOPE 内容。发现范围蔓延必须回退并在会话记录中说明。

## 5. 质量门槛 (Definition of Done)

一个功能/阶段被视为"完成"必须同时满足：

- [ ] 实现代码通过对应单元测试与集成测试
- [ ] 代码符合 [CODING_STANDARDS.md](03_CODING_STANDARDS.md)
- [ ] 不破坏任何既有测试（CI 全绿）
- [ ] 公共 API 有类型注解与文档注释
- [ ] [PROGRESS.md](05_PROGRESS.md) 状态已更新
- [ ] 未引入 OUT OF SCOPE 内容
- [ ] 若涉及架构变更，已更新 [ARCHITECTURE.md](02_ARCHITECTURE.md)

## 7. 关键决策记录 (ADR 摘要)

- **ADR-001**：选择 GDScript 而非 C#，保证单一语言栈与跨平台一致性。
- **ADR-002**：规则引擎做成纯逻辑层（非 Node），换取可测试性与未来可移植性。
- **ADR-003**：联机优先于 AI，因为联机架构一旦定型，AI 可作为"特殊客户端/服务器内 bot"接入，反之则需重构。
- **ADR-004**：几何色块美术，换取开发速度，渲染层抽象后可后续替换素材。
- **ADR-005**：服务器权威模型，避免客户端作弊与状态不一致，代价是延迟敏感操作需乐观反馈（卡坦为回合制，可接受）。
- **ADR-006**：GUT 锁定 9.4.0。9.5.0+ 引入 `Logger`↔`GutErrorTracker` 的 class_name 循环依赖，Godot 4.3 不支持循环依赖导致解析失败。9.4.0 是无该问题的最新版本。若后续升级到 Godot 4.4+（修复了循环依赖），可解锁 GUT 9.5.0+。

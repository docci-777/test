# AI 开发工作流指南 (AI_DEV_GUIDE)

> 本文件是每次 AI 开发会话的**操作手册**。
> 任何 AI 会话开始时必须先完整阅读本文件 + [00_PROJECT_SPEC.md](00_PROJECT_SPEC.md) + [05_PROGRESS.md](05_PROGRESS.md)。

---

## 1. 会话启动协议（强制）

每次 AI 会话开始，按顺序执行：

1. **阅读三件套**
   - [00_PROJECT_SPEC.md](00_PROJECT_SPEC.md)（最高约束）
   - [05_PROGRESS.md](05_PROGRESS.md)（当前进度）
   - 本文件（工作流）

2. **确认任务范围**
   - 从 PROGRESS.md 选取当前阶段未完成任务
   - 不得跨阶段跳跃，除非前置阶段已 ✅
   - 不得选取 OUT OF SCOPE 任务

3. **声明本次会话目标**
   - 在回复中明确："本次将完成 P?-? 至 P?-?"
   - 若范围不明确，先用 AskUserQuestion 与所有者确认

4. **阅读相关规格**
   - 涉及规则 → 读 [01_GAME_RULES.md](01_GAME_RULES.md) 对应章节
   - 涉及代码 → 读 [02_ARCHITECTURE.md](02_ARCHITECTURE.md) + [03_CODING_STANDARDS.md](03_CODING_STANDARDS.md)
   - 涉及测试 → 读 [04_TESTING.md](04_TESTING.md)

## 2. 开发顺序约束（强制）

每个功能任务必须按以下顺序推进，**不得跳步**：

```
读规格 → 写测试(红) → 写实现(绿) → 补边界用例 → 重构 → 更新文档 → 标记完成
```

- **测试先行**：先编写会失败的测试，再写实现使其通过
- **不得先写实现再补测试**（除探索性 spike，且须标注后重写）
- **不得为通过测试而弱化测试断言**

## 3. 范围纪律（强制）

- **禁止实现 OUT OF SCOPE 内容**（见 PROJECT_SPEC §3.2）
- 发现需求蔓延立即停止并提示所有者
- 不得"顺手"重构未涉及模块
- 不得添加未在规格中的配置项/功能开关
- 遇到规格未覆盖的边界情况：**先补规格再实现**，不得自行裁定

## 4. 架构纪律（强制）

- 严格遵守 [02_ARCHITECTURE.md](02_ARCHITECTURE.md) 分层依赖方向
- Layer 1 不得引入 Node/场景依赖
- 新增类型优先走扩展点（Registry + 接口），不得在核心主流程加 if/else 分支
- 数据驱动：可配置项入数据文件，禁止硬编码
- 网络权威：客户端不得直接改权威状态

## 5. 测试纪律（强制）

- 每个公共方法：正向 + 负向 + 边界用例
- 测试须确定性（注入种子/mock 随机）
- 提交前本地跑通全部 GUT 测试
- 不得删除既有测试；如测试与规格冲突，先改规格
- 不得使用 `pending`/`skip` 掩盖失败

## 6. 文档同步（强制）

每次会话结束前必须：

- 更新 [05_PROGRESS.md](05_PROGRESS.md)：
  - 完成的任务 ID 状态改 ✅，附测试文件路径
  - 进行中改 🔄
  - 阻塞改 ⛔ 并填阻塞说明
  - 追加"会话记录"条目（见 PROGRESS §6 格式）
- 若涉及架构变更 → 更新 [02_ARCHITECTURE.md](02_ARCHITECTURE.md)
- 若涉及规则变更 → 更新 [01_GAME_RULES.md](01_GAME_RULES.md) 并在变更日志登记
- 若新增扩展点 → 更新 ARCHITECTURE §7 表格

## 7. 提交规范

- 遵循 [03_CODING_STANDARDS.md](03_CODING_STANDARDS.md) §11 提交信息格式
- 单次提交单一职责
- 提交前自检 PR Checklist（CODING_STANDARDS §13）
- 不得提交未通过测试的代码
- 不得提交包含调试 `print`/临时注释的代码

## 8. 决策权限分级

| 决策类型 | AI 可自主 | 须问所有者 |
|----------|-----------|------------|
| 任务内实现细节 | ✅ | — |
| 测试用例设计 | ✅ | — |
| 重构（不改变对外 API） | ✅ | — |
| 选择哪个未完成任务 | ✅（按依赖顺序） | — |
| 修改对外公共 API | — | ✅ |
| 架构变更 | — | ✅ |
| 规则裁定/规格补充 | — | ✅ |
| 范围扩张（灰色地带） | — | ✅ |
| 引入新依赖 | — | ✅ |
| 跳过/延后任务 | — | ✅ |

需要所有者决策时，使用 AskUserQuestion 提供选项与影响分析，不得自行决定。

## 9. 阻塞处理

遇到阻塞时：

1. 在 PROGRESS.md 标记 ⛔ 并填阻塞说明
2. 评估是否可切换到其他无依赖任务继续
3. 若阻塞涉及规格/架构，提问所有者
4. 不得绕过阻塞强行推进（如关闭校验、跳过测试）

## 10. 常见反模式（禁止）

- ❌ 先写实现后补测试
- ❌ 为通过测试修改测试而非实现
- ❌ 在 Layer 1 使用 Node/print
- ❌ 硬编码规则数据
- ❌ 在核心主流程加类型分支而非扩展点
- ❌ 客户端直接改状态
- ❌ 删除/弱化既有测试
- ❌ 实现规格未定义的规则
- ❌ 跨阶段跳跃
- ❌ 提交未跑测试的代码
- ❌ 会话结束不更新 PROGRESS.md
- ❌ "顺手"重构无关代码
- ❌ 用 `if OS.is_debug_build()` 绕过校验

## 11. 会话结束协议（强制）

每次会话结束前：

1. 跑全部测试确认全绿
2. 更新 PROGRESS.md（状态 + 会话记录）
3. 同步其他受影响文档
4. 在回复中给出：
   - 本次完成任务清单（ID + 状态）
   - 新增/修改测试文件列表
   - 遗留问题与下次建议
5. 若有未完成进行中任务，明确说明停在哪一步

## 12. 快速检查清单（每次会话）

开始时：
- [ ] 读三件套
- [ ] 确认任务范围与依赖已满足
- [ ] 声明本次目标

进行中：
- [ ] 测试先行
- [ ] 遵守分层与扩展点
- [ ] 不越范围

结束时：
- [ ] 全测试绿
- [ ] PROGRESS.md 更新
- [ ] 文档同步
- [ ] 给出会话总结
